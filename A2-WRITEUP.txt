Problem 2: kmeans_serial vs kmeans_mpi
======================================

  Discuss how you chose to parallelize your serial version of K-means in
  the program `kmeans_mpi.c'. Answer the following questions briefly.

  1. How is the input and output data partitioned among the processors?

Our original serial code does everything in one process, meaning that it reads
the dataset, stores all the points, computes the cluster centers, and then
reassigns all the points. For the MPI version, processor 0 reads the full dataset and then 
distributes it across multiple processors using MPI_Scatterv. Because of this, instead of one 
loop going through every point, each processor only goes through its own local chunk in 
order to give the parallel version speedup.

  2. What communication is required throughout the algorithm?

Because the data is split up, the processors need to communicate with each other 
and have to agree on the cluster centers at each step. Each processor computes 
partial cluster sums using its own local points. Then, those partial 
results are combined to get the new global cluster centers. After that, processor 0 broadcasts
the updated centers back out to all processors so they can do the next assignment step.
Each processor also keeps track of how many assignments changed locally, and those 
local counts are combined to get the total number of changes. Once there are no more
global changes, the algorithm finishes.

  3. Which MPI collective communication operations did you employ?

The main MPI operations we use are MPI_Bcast, MPI_Scatterv, 
MPI_Reduce, MPI_Allreduce, and MPI_Gatherv. MPI_Bcast is used 
to send metadata and updated cluster centers from processor 0 to the other processors.
MPI_Scatterv is used to divide the dataset across processors. MPI_Reduce is used to combine partial 
feature sums and cluster counts on processor 0. MPI_Allreduce is used to combine the local 
assignment-change counts into a global total. At the end, MPI_Gatherv is used to collect the 
final assignments back to processor 0.

kmeans_mpi Discussion Questions
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

  Analyze your table of results and answer the following questions.
  1. Did using more processors result in speedups?
  Yes, using more processors resulted in large apparent speedups compared to the single-processor case. 
  The runtimes with one processor were significantly larger (3.35 s, 17.77 s, and 47.51 s for the three datasets), 
  while runs with two or more processors completed in around 0.07–0.14 seconds. 
  This shows that parallelization shortened runtime compared to the serial execution.

  2. Describe any trends or anomalies you see in the timings and
  speculate on their causes - e.g. was there are a steady increase in
  runtimes, steady decrease, or jagged changes in timing?
  The main trend is a large drop in runtime when moving from one processor to multiple processors. 
  After two processors, the runtimes stay almost constant at around 0.07 seconds regardless of processor count. 
  This might be because of communication overhead or timing resolution making differences between runs difficult to notice.
  
  3. Try to explain how number of processors and problem size seem to
  affect runtimes/speedup in the problem.
  Problem size clearly affects runtime with one processor, where larger datasets take longer to run. 
  With multiple processors, runtimes stay about the same across the datasets because the work is split across processors. 
  In general, larger problems benefit more from parallel processing than smaller ones.
